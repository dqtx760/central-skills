# 短视频文案可看性升级需求文档-执行版

## 1. 结论

本轮升级的核心不是让文案“更白话”，也不是机械强调“和普通人有关”，而是解决一个更底层的问题：

> 文案有信息，但没有观看欲。

因此，Skill 必须从“把 AI 热点写成文案”升级为“把 AI 热点筛选并改写成有停留理由的短视频口播”。

一条合格文案必须让用户在前 5-10 秒知道：

- 为什么这条内容值得看。
- 这件事有什么变化或新鲜点。
- 它和用户有什么关系。
- 看完能得到什么判断。

## 2. 本轮升级范围

### 2.1 要做

- 新增可看性四问。
- 新增内部可看性评分。
- 新增可看角度生成。
- 优化 Hook 生成规则。
- 优化正文节奏推进。
- 增加枯燥文案反向改写规则。
- 质量检查中新增“观看欲”标准。
- 同步更新 Skill 主文件、README 和压缩包。

### 2.2 不做

- 不改变 AI HOT 作为默认热点源。
- 不取消账号定制分支。
- 不改变 70-85 秒数字人口播定位。
- 不引入真人出镜或录屏实操。
- 不暴露底层方法名和内部来源。
- 不为了反差编造冲突。
- 不为了去 AI 味编造故事、数据或个人经历。

## 3. 默认用户与内容定位

默认面向：

> 对 AI 有兴趣，但没有耐心看行业简报的短视频用户。

这个用户不一定完全不懂 AI，但他不会为枯燥信息停留。  
他关心的是：

- 这件事有什么变化？
- 它为什么现在值得看？
- 它会影响谁？
- 我会不会误判它？
- 看完这 80 秒，我能得到什么判断？

默认不面向：

- 开发者。
- 模型研究者。
- 技术媒体读者。
- 行业投资人。
- 只关心参数和论文的人。

除非用户明确要求账号定制，否则不要默认写给这些专业人群。

## 4. 选题筛选升级

### 4.1 可看性四问

每个热点进入文案生成前，必须先回答：

| 问题 | 作用 |
|---|---|
| 这件事有什么变化？ | 避免只复述新闻 |
| 这件事有什么可看点？ | 找到停留理由 |
| 这件事和用户有什么关系？ | 建立观看动机 |
| 看完能得到什么判断？ | 提供信息回报 |

如果四个问题里只能回答一个，默认跳过，不强行生成。

### 4.2 内部可看性评分

评分用于内部筛选，不建议完整输出给用户，避免结果像评审表。

| 维度 | 分值 | 判断 |
|---|---:|---|
| 停留理由 | 0-2 | 前 5 秒有没有继续看的理由 |
| 新鲜变化 | 0-2 | 有没有明确变化、反差或趋势 |
| 用户关联 | 0-2 | 能不能落到用户关心的场景 |
| 信息回报 | 0-2 | 看完能不能得到一个判断 |
| 表达空间 | 0-2 | 能不能用口播和 AI 画面讲清楚 |

处理规则：

- 8-10 分：优先生成。
- 6-7 分：可以生成，但必须强化 Hook 和角度。
- 5 分及以下：默认跳过或换选题。

用户可见输出只保留：

```markdown
- 可看点：{为什么这条热点值得做成短视频}
```

## 5. 可看角度生成

每条入选热点必须先确定可看角度，再写文案。

输出字段：

```markdown
- 表面事件：{发生了什么}
- 可看点：{为什么值得看}
- 用户关联：{和用户有什么关系}
- 核心判断：{看完应该得到什么判断}
- 5 秒 Hook：{停留理由}
```

注意：

- 冲突和反差是优先抓手，但不是唯一抓手。
- 可以用利益、风险、场景代入、新鲜变化、明确判断制造可看性。
- 不为了反差制造假冲突。

## 6. Hook 规则升级

Hook 的任务不是介绍新闻，而是制造停留。

### 6.1 合格 Hook

至少满足两项：

- 有关系：这件事可能影响用户。
- 有利益：用户可能少做某类重复工作。
- 有风险：用户可能误判或错过变化。
- 有新鲜变化：以前怎样，现在怎样。
- 有悬念：真正值得看的点还没说完。
- 有判断：先给一个明确观点，再讲事实。

### 6.2 禁止型开头

默认避免：

```text
某某公司发布了最新 AI 模型。
今天我们来聊一个 AI 热点。
AI 行业又迎来了重大变化。
最近 AI 圈发生了一件大事。
```

这些开头的问题是：用户不知道为什么要继续看。

### 6.3 推荐型开头

```text
这条 AI 新闻，如果只看模型名字，你可能会错过真正重要的变化。
```

```text
如果你经常让 AI 帮你写资料，这次更新真正该看的不是发布会。
```

```text
它表面是一次功能更新，真正影响的是普通人使用 AI 的门槛。
```

## 7. 正文结构升级

推荐结构：

1. 0-5 秒：先给停留理由。
2. 5-15 秒：交代事实，只讲必要背景。
3. 15-30 秒：讲清表面看点和真正看点。
4. 30-55 秒：落到用户场景。
5. 55-75 秒：给出判断。
6. 75-85 秒：用一句有力的话收束。

正文不能只按：

```text
发生了什么 -> 为什么重要 -> 有哪些影响 -> 总结一下
```

要改成：

```text
为什么要停留 -> 事情是什么 -> 真正看点是什么 -> 和用户有什么关系 -> 现在该怎么看
```

## 8. 枯燥文案反向改写规则

当文案像新闻稿、行业简报或 AI 总结时，按以下规则改：

| 枯燥类型 | 改写方向 |
|---|---|
| 新闻复述 | 改成“这件事真正影响谁” |
| 技术介绍 | 改成“用户能少做什么、多得到什么” |
| 行业简报 | 改成“这件事暴露了什么变化” |
| 平铺直叙 | 改成“先给判断，再讲事实” |
| 术语堆叠 | 改成“一句术语 + 一句白话解释” |
| 空泛总结 | 改成“一个明确判断或提醒” |
| 工具说明书 | 改成“使用场景 + 适合谁/不适合谁” |

如果改完仍然枯燥，换选题，不继续硬写。

## 9. 表达要求

文案要像数字人能直接念出来的话：

- 句子短。
- 信息密度稳。
- 不端着讲。
- 不像报告。
- 不像新闻稿。
- 不像工具说明书。
- 不连续堆叠专业词。
- 不用空泛大词撑气势。

优先删除：

- 未来可期。
- 值得关注。
- 意义重大。
- 不可否认。
- 从某种程度上说。
- 这不仅是技术进步，更是时代趋势。
- 今天我们来看看。
- 最近很多人都在讨论。

必须保留：

- 可追溯事实。
- 具体变化。
- 用户关联。
- 核心判断。
- 口播节奏。

## 10. 字幕要求

字幕按口播节奏切，不按书面段落切。

- 一句一行。
- 每行通常 8-18 个中文字。
- 单行尽量不超过 24 个中文字。
- 一行只表达一个清楚意思。
- 不用项目符号。
- 可直接复制到剪辑软件。

## 11. 质量检查

每条输出前必须检查：

| 检查项 | 通过标准 |
|---|---|
| 停留理由 | 前 5 秒知道为什么继续看 |
| 可看点 | 不只是新闻复述 |
| 新鲜变化 | 有变化、反差、趋势、利益或风险 |
| 用户关联 | 明确落到用户场景 |
| 信息回报 | 看完能得到一个判断 |
| 口播友好 | 能直接念出来 |
| 不枯燥 | 不像行业简报、新闻稿、报告 |
| 事实可靠 | 不编造来源、数据和产品能力 |
| 字幕可用 | 一句一行，方便剪辑 |

不达标处理：

1. 先换 Hook。
2. 再换切入角度。
3. 再压缩表达。
4. 仍然枯燥就换选题。

## 12. 输出格式调整

每条卡片新增用户可见字段：

```markdown
- 可看点：{为什么这条热点值得做成短视频}
- 用户关联：{和目标用户有什么关系}
- 核心判断：{看完这条视频应该得到什么判断}
```

内部可看性评分不强制输出，除非用户要求看筛选过程。

## 13. 验收标准

升级完成后，Skill 生成的每条文案必须满足：

- 有可追溯 AI 热点来源。
- 前 5 秒有停留理由。
- 不只是新闻复述。
- 有新鲜变化、反差、利益、风险或明确判断。
- 明确回答和用户有什么关系。
- 看完能得到一个判断。
- 表达不像报告、新闻稿或 AI 总结。
- 不编造事实、数据、来源和产品能力。
- 字幕一句一行，可直接用于剪辑。
